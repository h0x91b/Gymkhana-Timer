# Техническое задание: Gymkhana Timer (PWA)

## Идея

Приложение для замера времени прохождения **маршрута** джимханы на мотоцикле. Маршрут — это трасса, в которую заезжаешь и из которой выезжаешь через одну и ту же линию (стартовую/финишную); это **не круг** (lap), термин `lap` в документации и в коде не используется. Линия обозначена четырьмя конусами. Телефон устанавливается на штатив напротив этой линии. Приложение через камеру детектирует момент пересечения мотоциклом линии:

- Первое пересечение → старт таймера
- Второе пересечение → стоп таймера, показ результата

Телефон во время заезда **не двигается** — это ключевое упрощение, на котором строится весь алгоритм.

### Hands-free session mode (основной режим эксплуатации)

Главный инвариант UX: **после начального ручного сетапа (камера + ROI) райдер НЕ подходит к телефону между заездами.** Весь цикл «заезд → результат → подготовка к следующему заезду» происходит автоматически.

Последовательность одной сессии:

1. Райдер один раз делает сетап: даёт доступ к камере, размечает ROI, садится на мотоцикл. Отдельной кнопки `Start session` нет — как только ROI зафиксирован, приложение сразу входит в hands-free-цикл.
2. Приложение входит в `OBSERVING` — ждёт, когда ROI будет пустым и стабильным ≥ 2 секунды, и берёт свежий reference frame. Фон экрана **оранжевый** (`--signal-wait`) — «ещё не готов».
3. Переходит в `ARMED` — голос: `ready to go`. Фон **насыщённо-зелёный** (`--signal-go`). На экране — крупно последнее время маршрута (или `0.000`, если это первый заезд в сессии).
4. Райдер пересекает линию → `RUNNING`, голос `start`, таймер тикает крупно. Прошлое время показывается маленьким под крупным. Фон — нейтральный parchment.
5. Второе пересечение (после cooldown) → `FINISHED`, голос `finish, 42 point 3`, большое время остаётся на экране.
6. Включается **10-секундный обратный отсчёт** — отображается внизу меньшим размером (`9 … 8 …`). Большое время на таймере — последний результат. За эти 10 сек райдер успевает подъехать обратно и оценить результат визуально. Фон — оранжевый.
7. По истечении 10 сек → снова `OBSERVING` → как только ROI стабилен, голос `ready to go`, идём на шаг 3.

Если `OBSERVING` продолжается > 20 сек без стабилизации — фон становится **красным** (`--signal-error`). Пока мы не в `ARMED` (т.е. в `OBSERVING` или в `ERROR`-визуале), каждые **15 секунд** голос произносит `not ready` — чтобы райдер, не глядя на телефон, понимал, что система всё ещё не готова принять старт.

Сессия останавливается только вручную (тап по экрану → появляется кнопка **Stop session**). Это единственный момент после сетапа, когда нужно подойти к телефону.

Правила отображения:

- **Таймер — доминирующий элемент экрана всегда.** Занимает ~50vh по высоте, центр, serif, tabular-nums.
- Во время `RUNNING` — тикает текущее время, прошлый результат виден маленьким рядом/снизу.
- Во время `FINISHED`, 10s countdown, `OBSERVING`, `ARMED` — таймер показывает **последнее зафиксированное время маршрута** (не обнуляется до следующего старта).
- 10-секундный countdown — мелкий вторичный индикатор под большим временем, считает в обратном порядке.
- После завершения countdown и перехода в `ARMED` — подпись `Ready to go` вместо countdown.

## Формат приложения: PWA

Приложение реализуется как **Progressive Web App** — веб-приложение с манифестом и service worker'ом, которое устанавливается на Android как нативное.

**Почему PWA, а не нативный Android:**
- Быстрая разработка и итерация
- Кросс-платформенность (Android, iOS, планшеты)
- Не нужен Play Store, подпись APK, Gradle
- Обновления доставляются через обычный деплой
- Для этой задачи производительности веба достаточно с запасом

**Что PWA даёт:**
- Установка на домашний экран с иконкой
- Запуск в fullscreen без UI браузера
- Работа офлайн (на заезде может не быть интернета)
- Принудительная landscape-ориентация
- Wake Lock — экран не гаснет

## Основной алгоритм: Background Subtraction

Телефон неподвижен → фон статичен → любое движение в ROI (области между конусами) детектируется через разницу с референсным кадром.

### Логика покадрово

```
1. При старте "armed" режима — захватываем референсный кадр (или усреднённый из N первых кадров)
2. На каждом новом кадре:
   a. Берём только ROI (область между 4 конусами, заданная пользователем)
   b. Считаем per-pixel разницу с референсным кадром (|current - reference|)
   c. Бинаризуем по порогу → motion mask
   d. Считаем долю "движущихся" пикселей
3. Если доля превысила threshold (например, 15-20%):
   - Фиксируем timestamp кадра
   - Переходим в cooldown (игнорируем триггеры ~1-2 сек, чтобы не словить финиш сразу же)
4. State machine:
   IDLE → (первое срабатывание) → RUNNING → (второе срабатывание) → FINISHED
```

### Оптимизации

- **Downscale ROI** до 200-300 пикселей по большей стороне — считается мгновенно
- **Читать только ROI** через `getImageData(x, y, w, h)`, а не весь кадр
- **Морфологическая эрозия** (опционально) — чтобы убрать шум и мелкие тени
- **Плавающий reference frame** — медленно адаптировать референс под изменение освещения (skip если не будет проблем)

## Технологический стек

- **HTML + vanilla JavaScript** — без фреймворков, проект слишком маленький
- **`getUserMedia`** — доступ к камере
- **`requestVideoFrameCallback`** — колбэк на каждый кадр с точным timestamp'ом (`mediaTime` в секундах с субмиллисекундной точностью)
- **`<canvas>` + `getImageData`** — чтение пикселей для diff
- **`performance.now()`** — как бэкап для таймера
- **Wake Lock API** — блокировка засыпания
- **Service Worker** — офлайн-режим
- **Web App Manifest** — установка как приложение
- **`MediaRecorder`** (опционально) — запись видео заезда с оверлеем таймера
- **`speechSynthesis`** — голосовой фидбек ("старт", "финиш, 42.3 секунды")
- **`localStorage` или IndexedDB** — история заездов

## Функциональные требования

### MVP

1. **Запрос доступа к камере** при первом запуске
2. **Видео-превью на весь экран** в landscape
3. **Настройка ROI**: пользователь тапает 4 точки на превью (углы области между конусами). Можно прямоугольник для простоты начала, полигон — как улучшение.
4. **Настройка чувствительности**: слайдер для порога motion detection (или автокалибровка)
5. **Автоматический старт сессии**: как только ROI зафиксирован, приложение сразу входит в hands-free цикл — `OBSERVING → ARMED → RUNNING → FINISHED → 10s countdown → OBSERVING → …`. Отдельной кнопки `Start session` нет; `Stop session` доступен по тапу в экран.
6. **Детекция старта**: первое срабатывание фиксирует T0
7. **Таймер на экране**: огромные serif-цифры (~50vh), доминирующий элемент на всех экранах кроме сетапа
8. **Детекция финиша**: второе срабатывание фиксирует T1, показывает `T1 - T0`, результат остаётся на экране до следующего старта
9. **Голосовое оповещение**: `start`, `finish, N seconds`, `ready to go`, `not ready` (каждые 15 сек пока не в `ARMED`)
10. **Auto re-arm**: 10-секундный countdown мелко под таймером → автоматический возврат в `OBSERVING → ARMED`. Ручного Reset между заездами нет. Stop session останавливает цикл по тапу.

### Желательно

- Сохранение истории заездов (время + дата)
- Отображение лучшего времени сессии
- Запись видео параллельно с оверлеем таймера
- Отображение FPS и статуса детектора на экране (для отладки)

### PWA-инфраструктура

- `manifest.json` с `display: "fullscreen"`, `orientation: "landscape"`
- Иконки 192x192 и 512x512

## Подводные камни, которые надо учесть

1. **Точность таймера**
   - Использовать `timestamp` **из самого кадра** (параметр `metadata.mediaTime` в `requestVideoFrameCallback`), а не `performance.now()` в момент обработки — иначе поплывёт от jitter'а обработки
   - FPS потолок в вебе обычно 30, запросить 60 через constraints (`frameRate: { ideal: 60 }`), но гарантировать нельзя
   - Квантование на 30 FPS = 33 мс — для джимханы ок

2. **Ложные срабатывания**
   - Тени от байка при низком солнце — особенно вечером
   - Другие люди/мотоциклы/вороны в кадре
   - Изменение освещения (облако наехало)
   - Решение: ROI узкий, threshold подобран, cooldown после срабатывания, опционально — ML Kit confirm в v2

3. **Браузерные ограничения**
   - Tab throttling при сворачивании — решается Wake Lock + fullscreen + "не трогать телефон во время заезда"
   - Разрешения на камеру в PWA запоминаются, но при первом запуске надо запросить через user gesture

4. **`getImageData` на всём кадре — медленно**. Всегда читать только ROI.

5. **Ориентация камеры**. В landscape-режиме проверить, что оси X/Y ROI совпадают с тем, что видит камера.

## State Machine детектора (hands-free session mode)

```
                                              Stop session (user tap)
                ┌─────────────────────────────────────────────────────────┐
                │                                                         │
                ▼                                                         │
           ┌─────────┐    ROI picked      ┌────────────┐  ROI stable ≥2s  │
           │  IDLE   │ ─────────────────> │ OBSERVING  │ ───────────────> │
           └─────────┘                    └────────────┘                  │
                ▲                               ▲                          │
                │                               │                    ┌───────┐
                │                      10s done │                    │ ARMED │
                │                               │                    └───┬───┘
                │                          ┌─────────┐                   │
                │                          │ COOLDOWN│                   │ motion!
                │                          │  (10s)  │                   │ (first cross)
                │                          └─────────┘                   │
                │                               ▲                        ▼
                │                               │                   ┌─────────┐
                │                               │ motion! (second,  │ RUNNING │
                │                               │ after cooldown)   └────┬────┘
                │                          ┌────┴────┐                   │
                └──── (never auto) ──────  │FINISHED │ <─────────────────┘
                                           └─────────┘
```

Ключевые отличия от «кнопочного» режима:
- `OBSERVING` ждёт **пустого, стабильного** ROI и сам захватывает свежий reference (компенсация дрейфа освещения между заездами).
- `FINISHED → COOLDOWN(10s) → OBSERVING` происходит автоматически, без участия райдера.
- `IDLE` — только состояние до первого ROI и после ручного `Stop session`. Переход `IDLE → OBSERVING` срабатывает сам, как только ROI зафиксирован.

## Приоритеты разработки

1. **Этап 1 (MVP для проверки идеи)**: камера + ROI прямоугольником + background subtraction + таймер + голос. Без PWA-обвеса.
2. **Этап 2 (оформление как PWA)**: manifest, service worker, иконки, офлайн-кэш, Wake Lock.
3. **Этап 3 (полировка)**: история заездов, запись видео, автокалибровка чувствительности, красивый UI.
4. **Этап 4 (если понадобится)**: ML-подтверждение через TensorFlow.js / ML Kit для фильтрации ложных срабатываний.

## Не нужно

- OpenCV.js — избыточно, всё считается вручную на canvas
- Никаких фреймворков (React/Vue/etc) — слишком маленький проект
- Никакого бэкенда — всё клиентское
- Никаких аккаунтов и логинов
- Никакой аналитики/телеметрии

## Критерий успеха

Приложение стабильно замеряет время на реальной тренировке джимханы при нормальном освещении с точностью до десятых секунды, не требует возни с настройками между заездами (один раз выставил ROI — катаешь).
